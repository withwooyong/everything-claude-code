# Everything Claude Code (ECC) 실전 사용 가이드

> 대상: React/Next.js + Java/Python 개발자, Kotlin/Thymeleaf 레거시 리팩토링 포함
> 작성일: 2026-03-30

---

## 0. 빠른 시작: 자연어로 계획 → 수정 → 팀 병렬 작업

ECC를 처음 쓴다면 이 섹션만 읽으면 됩니다. **슬래시 커맨드를 외울 필요 없이**, 아래처럼 자연어로 말하면 됩니다.

### Step 1: 작업계획서 생성

```
사용자 관리 API를 만들려고 해.
요구사항은 다음과 같아:
- 회원가입/로그인 (JWT 인증)
- 프로필 CRUD
- 비밀번호 변경
- 백엔드: Spring Boot + JPA
- 프론트: Next.js

구현 계획서 만들어줘.
```

→ Claude가 Phase별 계획서를 작성하고 **CONFIRM 대기**

### Step 2: 계획서 수정 (마음에 안 드는 부분을 그냥 말하면 됩니다)

```
Phase 2에서 소셜 로그인(구글, 카카오)도 추가해줘.

비밀번호 변경은 Phase 3으로 빼고, Phase 2는 핵심 기능에만 집중하자.

프론트 상태관리는 Zustand 대신 React Query만 쓰는 걸로 바꿔줘.

DB는 PostgreSQL로 하고, Redis 세션 저장소도 계획에 넣어줘.
```

→ Claude가 계획서를 수정해서 다시 보여줌

```
좋아, 이 계획서로 확정.
```

### Step 3: 팀 병렬 작업 요청

**기본형** — 가장 많이 쓰게 될 패턴:
```
Phase 1을 팀 만들어서 병렬로 진행해줘.
```
→ Claude가 자동으로 작업을 분배하고 에이전트 팀 구성

**역할 지정형** — 더 구체적으로 제어하고 싶을 때:
```
Phase 1을 팀 만들어서 병렬로 진행해줘.
- 백엔드팀: Spring Boot 엔티티 + Repository + API 컨트롤러
- 프론트팀: 로그인/회원가입 페이지 + API 연동
- 테스트팀: 백엔드 단위테스트 + API 통합테스트
```

**조건 추가형** — 품질 기준을 명시하고 싶을 때:
```
Phase 1을 팀 만들어서 병렬로 진행해줘.
조건:
- TDD로 진행 (테스트 먼저)
- 백엔드 API는 OpenAPI 스펙 먼저 정의
- 프론트는 백엔드 API 스펙 기준으로 개발
- 각 팀 작업 끝나면 코드리뷰까지 완료
```

### Step 4: 진행 중 추가 요청

```
# 결과 확인
각 팀 작업 결과 요약해줘.

# 문제 수정
백엔드팀 결과에서 JWT 토큰 만료 처리가 빠져있네. 추가해줘.

# 추가 병렬 작업
Phase 2도 같은 방식으로 팀 만들어서 병렬로 진행해줘.

# 통합 테스트
Phase 1, 2 통합해서 전체 E2E 테스트 돌려줘.
```

### Step 5: 마무리 한 문장

```
전체 코드리뷰 하고, 문제 없으면 커밋하고 PR 만들어줘.
문서 정리하고 핸드오프도 해줘.
```

### 실전 풀 시나리오 (복사해서 바로 쓸 수 있는 형태)

```
── 세션 시작 ──
/resume-session

── 계획 ──
bearwatch 프로젝트에 실시간 알림 기능을 추가하려고 해.
요구사항:
- 주가 급등/급락 시 웹소켓으로 실시간 푸시
- 사용자별 알림 설정 (종목, 등락률 임계값)
- 알림 히스토리 조회 API
- 프론트: 알림 벨 아이콘 + 드롭다운 패널
구현 계획서 만들어줘.

── 수정 ──
웹소켓 대신 SSE(Server-Sent Events)로 변경해줘.
알림 히스토리는 Phase 2로 빼자.

── 확정 + 병렬 실행 ──
좋아 확정. Phase 1을 팀 만들어서 병렬로 TDD 기반으로 진행해줘.

── 결과 확인 + 보완 ──
각 팀 결과 요약해줘.
SSE 연결 끊김 시 재연결 로직이 빠져있네. 추가해줘.

── 마무리 ──
코드리뷰 하고 커밋, PR 만들어줘. 핸드오프도.
```

> 핵심: **"팀 만들어서 병렬로 진행해줘"** 한 문장이면 Claude가 에이전트를 자동 분배합니다.
> 역할이나 조건을 붙이면 더 정밀하게 제어할 수 있습니다.

## 1. 현재 설치 상태 확인

당신의 환경에는 이미 ECC 플러그인이 설치되어 있습니다:

```json
// ~/.claude/settings.json
"enabledPlugins": {
  "everything-claude-code@everything-claude-code": true
}
```

**이것만으로 59개 슬래시 커맨드, 20+ 전문 에이전트, 100+ 스킬이 자동 활성화됩니다.**
별도로 스킬 명령어 환경으로 전환할 필요 없이, 평소처럼 자연어로 요청하면 됩니다.

## 2. 핵심 개념: "스킬 명령어를 외워야 하나요?"

### 답변: 아니요, 자연어로 충분합니다

ECC의 스킬/커맨드는 **Claude가 자동으로 인식하고 활성화**합니다. 예를 들어:

| 당신이 하던 대로 말하면 | ECC가 내부적으로 활성화하는 것 |
|---|---|
| "작업계획서 만들어줘" | `/plan` 스킬 → 리스크 평가 + 단계별 계획 |
| "팀 만들어서 병렬로 작업해줘" | `/devfleet` 또는 Agent 팀 모드 (worktree 분리) |
| "테스트 먼저 작성하고 개발해줘" | `/tdd` 스킬 → RED-GREEN-REFACTOR 사이클 |
| "코드리뷰 해줘" | `/code-review` → 보안/품질/유지보수성 검토 |
| "보안 검토해줘" | `security-reviewer` 에이전트 → OWASP Top 10 점검 |
| "문서 정리해줘" | `/update-docs` → 코드맵/README 동기화 |
| "핸드오프 해줘" | `/handoff` → CHANGELOG + HANDOFF.md 생성 |
| "PR 만들어줘" | git push + `gh pr create` → 리뷰 요청 |
| "푸시해줘" | git push (ECC 훅이 커밋 품질+시크릿 자동 검증) |

### 슬래시 커맨드를 직접 쓰면 더 좋은 경우

더 **정밀한 제어**가 필요할 때만 슬래시 커맨드를 직접 씁니다:

```
/plan                    ← 구현 전 반드시 확인받고 싶을 때
/tdd                     ← TDD 사이클을 엄격하게 강제하고 싶을 때
/context-budget          ← 컨텍스트 사용량 확인하고 싶을 때
/save-session            ← 세션 종료 전 맥락 저장
/resume-session          ← 다음 세션에서 이어서 작업
```

## 3. 보안은 어느 단계에서 처리되나?

### 답변: 3중으로 자동 처리 + 필요시 수동 점검

ECC는 보안을 **별도 단계가 아니라 개발 전 과정에 녹여서** 처리합니다.

### 자동 보안 (요청 없이 동작)

#### Layer 1: 코드 작성 시점 — 에이전트 자동 개입
`security-reviewer` 에이전트는 **PROACTIVELY(능동적)** 로 설정되어 있어서,
인증, 사용자 입력 처리, API 엔드포인트, 결제 등 민감한 코드를 작성하면
**요청하지 않아도 자동으로 보안 검토를 수행**합니다.

자동 감지 대상:
- 하드코딩된 API 키, 비밀번호, 토큰
- SQL 인젝션, XSS 취약점
- 인증/인가 누락
- 안전하지 않은 역직렬화
- SSRF, 경로 탐색 공격

#### Layer 2: 커밋 시점 — ECC 훅 자동 차단
`pre-bash-commit-quality` 훅이 커밋 전에 자동으로:
- **시크릿 감지** (API 키, 패스워드 등 소스에 포함된 비밀 정보)
- **console.log/debugger 잔류** 확인
- **커밋 메시지 형식** 검증
- **lint** 스테이지된 파일 검사

시크릿이 발견되면 **커밋 자체가 차단**됩니다.

#### Layer 3: 코드리뷰 시점 — 보안 항목 포함
`/code-review`를 실행하면 품질 검토와 함께 보안 항목이 자동 포함됩니다:
- CRITICAL: 하드코딩된 자격증명, SQL 인젝션, XSS, 인증 누락
- HIGH: 입력 미검증, 취약한 의존성, 경로 탐색
- CRITICAL/HIGH 이슈가 있으면 **커밋 승인 거부**

### 수동 보안 (명시적으로 요청할 때)

특별히 심층 보안 점검이 필요한 경우에만 직접 요청합니다:

```
# 전체 보안 스캔
보안 리뷰 해줘

# 또는 슬래시 커맨드로
/security-scan          ← AgentShield 기반 전체 스캔 (1282개 규칙)
```

### 결론: 평소처럼 개발하면 됩니다

```
                  자동 보안이 개입하는 지점
                  ━━━━━━━━━━━━━━━━━━━━━━━

  코드 작성 ──→ [Layer 1: security-reviewer 자동 감지]
      ↓
  테스트 ────→ (TDD로 엣지 케이스 커버)
      ↓
  코드리뷰 ──→ [Layer 3: /code-review에 보안 항목 포함]
      ↓
  커밋 ──────→ [Layer 2: 훅이 시크릿/품질 자동 차단]
      ↓
  PR ────────→ (리뷰어가 보안 이슈 추가 확인)
      ↓
  푸시
```

인증, 결제, 파일 업로드 같은 **고위험 기능을 개발할 때만** "보안 리뷰도 꼼꼼히 해줘"라고 한 마디 추가하면 충분합니다.

## 4. 코드리뷰와 PR은 어느 단계에 넣나?

### 권장 워크플로우: 코드리뷰 → 커밋 → PR 순서

```
개발 완료
  ↓
코드리뷰 ← "코드리뷰 해줘" (보안+품질+유지보수성 통합 검토)
  ↓         CRITICAL/HIGH 이슈 있으면 수정 후 재검토
  ↓
커밋     ← "커밋해줘" (ECC 훅이 시크릿/린트 자동 검증)
  ↓
PR 생성  ← "PR 만들어줘" (변경사항 요약 + 테스트 계획 자동 작성)
  ↓
푸시     ← "푸시해줘"
```

### `/code-review`가 검사하는 것

| 카테고리 | 검사 항목 | 심각도 |
|---------|----------|--------|
| 보안 | 하드코딩된 자격증명, SQL/XSS 인젝션, 인증 누락 | CRITICAL |
| 보안 | 입력 미검증, 취약 의존성, 경로 탐색 | HIGH |
| 품질 | 50줄 초과 함수, 800줄 초과 파일, 4단계 초과 중첩 | HIGH |
| 품질 | 에러 핸들링 누락, console.log 잔류, TODO/FIXME | HIGH |
| 모범사례 | 뮤테이션 패턴, 새 코드에 테스트 누락, 접근성(a11y) | MEDIUM |

### 자연어 요청 예시

```
# 코드리뷰만
코드리뷰 해줘.

# 코드리뷰 + 커밋 + PR 한 번에
코드리뷰 하고, 문제 없으면 커밋하고 PR 만들어줘.

# 특정 파일만 리뷰
src/api/payment.ts 보안 리뷰 해줘.
```

## 5. 기존 프로젝트에 ECC 적용하기

### 예시: Kotlin/Thymeleaf 레거시 → React/Spring Boot 리팩토링

#### Step 1: 프로젝트 디렉토리에서 Claude Code 시작
```bash
cd ~/projects/my-legacy-app
claude
```

#### Step 2: 계획 수립 (자연어로)
```
이 프로젝트는 Kotlin + Thymeleaf 기반 레거시야.
React + Spring Boot로 리팩토링하고 싶어.
현재 코드 분석하고 마이그레이션 계획서 만들어줘.
```

→ ECC의 `/plan` 스킬이 자동 활성화되어:
- 현재 코드베이스 분석
- 리스크 평가
- 단계별 마이그레이션 계획 작성
- **사용자 확인(CONFIRM) 후에만 코드 수정 시작**

#### Step 3: 팀 병렬 작업 요청
```
계획서 Phase 1을 팀 만들어서 병렬로 진행해줘.
- 백엔드: Spring Boot API 마이그레이션
- 프론트: React 컴포넌트 구현
- 테스트: 기존 기능 테스트 작성
```

→ ECC가 Agent 팀을 구성하여 git worktree로 격리된 환경에서 병렬 작업:
- 각 에이전트가 독립된 브랜치에서 작업
- 메인 컨텍스트 오염 없음
- 작업 완료 후 결과 통합

#### Step 4: 테스트 검증
```
마이그레이션된 API 엔드포인트 테스트해줘.
기존 Thymeleaf 페이지와 동일한 동작하는지 확인.
```

#### Step 5: 코드리뷰 (보안 포함)
```
코드리뷰 해줘. 인증 관련 코드는 보안 리뷰도 꼼꼼히.
```
→ `/code-review` + `security-reviewer` 자동 활성화
→ CRITICAL/HIGH 이슈 있으면 수정 후 재검토

#### Step 6: 커밋 + PR
```
문제 없으면 커밋하고 PR 만들어줘.
```
→ ECC 훅이 시크릿/린트 자동 검증 후 커밋
→ PR에 변경사항 요약 + 테스트 계획 자동 작성

#### Step 7: 문서 정리 & 핸드오프
```
작업 내용 문서 정리하고 핸드오프 해줘.
```

## 6. 신규 프로젝트에 ECC 적용하기

### 예시: Next.js + Python FastAPI 풀스택 프로젝트

#### Step 1: 프로젝트 생성부터
```bash
mkdir ~/projects/new-saas && cd ~/projects/new-saas
git init
claude
```

#### Step 2: 스펙 문서 기반 개발
```
SPEC.md 파일을 기준으로 개발할거야.
(SPEC.md 내용 또는 요구사항 설명)

구현 계획서 만들어줘.
```

#### Step 3: TDD 기반 백엔드 개발
```
계획서 확인했어. 백엔드 API부터 TDD로 개발해줘.
테스트 먼저 작성하고, 통과하는 코드 구현해.
```

→ ECC `/tdd` 워크플로우 자동 적용:
1. 인터페이스/스키마 정의
2. 실패하는 테스트 작성 (RED)
3. 최소한의 코드로 통과 (GREEN)
4. 리팩토링 (REFACTOR)
5. 각 단계별 Git 체크포인트 커밋

#### Step 4: 프론트엔드 병렬 개발
```
백엔드 API가 준비됐으니,
프론트엔드 팀을 만들어서 병렬로 개발해줘.
- 컴포넌트 구현
- API 연동
- E2E 테스트
```

#### Step 5: 코드리뷰 + 보안 + 스펙 검증
```
전체 코드리뷰 해주고, SPEC.md 기준으로 구현이 맞는지 검증해줘.
API 인증과 입력 검증 부분은 보안도 꼼꼼히 봐줘.
```
→ 코드리뷰(품질) + 보안 리뷰 + 스펙 일치 검증이 한 번에 수행

#### Step 6: 커밋 + PR
```
문제 없으면 커밋하고 PR 만들어줘.
```

#### Step 7: 문서 정리 & 핸드오프
```
문서 정리하고 핸드오프 해줘.
```

## 7. 컨텍스트 용량 관리 (맥락 유지 전략)

### 문제: AI Context 제한으로 긴 작업 중 맥락을 잃음

### ECC 해결책:

#### (1) 세션 저장/복원
```
# 작업 중간에 컨텍스트가 부족해질 것 같을 때
/save-session

# 새 세션에서 이어서 작업
/resume-session
```

#### (2) 팀 모드로 컨텍스트 분리
병렬 에이전트는 각각 독립된 컨텍스트를 가지므로, 메인 세션의 컨텍스트를 보존합니다.

```
팀을 만들어서 병렬로 진행해줘.
메인 세션은 전체 진행상황 관리만 하고,
실제 구현은 서브 에이전트가 처리해줘.
```

→ 메인 에이전트: 오케스트레이션 + 결과 취합만 담당
→ 서브 에이전트: 각자 독립 컨텍스트에서 구현 작업

#### (3) 컨텍스트 예산 확인
```
/context-budget
```
→ 현재 컨텍스트 사용량 분석, 최적화 방안 제시

#### (4) 핸드오프로 세션 간 연결
```
/handoff
```
→ HANDOFF.md에 현재 작업 상태, 다음 할 일, 주의사항 기록
→ 다음 세션에서 이 파일을 참조하여 맥락 복원

## 10. 언어별 자동 활성화되는 에이전트

| 프로젝트 언어 | 코드리뷰 에이전트 | 빌드 수정 에이전트 | 보안 에이전트 | TDD 에이전트 |
|---|---|---|---|---|
| React/Next.js (TS) | typescript-reviewer | build-error-resolver | security-reviewer | tdd-guide |
| Java/Spring Boot | java-reviewer | java-build-resolver | security-reviewer | tdd-guide |
| Python/FastAPI | python-reviewer | - | security-reviewer | tdd-guide |
| Kotlin | kotlin-reviewer | kotlin-build-resolver | security-reviewer | tdd-guide |

→ **파일 확장자와 프로젝트 구조를 보고 자동으로 적합한 에이전트가 선택됩니다.**
→ **`security-reviewer`는 모든 언어에서 공통으로 자동 활성화됩니다.**

## 12. ECC 설치 추가 세팅 (선택사항)

현재 플러그인은 설치되어 있지만, 프로젝트별 규칙을 추가로 설치하고 싶다면:

```bash
# ECC 저장소 디렉토리에서
cd ~/antigravity/everything-claude-code

# Java + Python + TypeScript 규칙 설치 (홈 디렉토리에)
./install.sh java python typescript

# 특정 프로젝트에만 설치하고 싶다면
# 해당 프로젝트 CLAUDE.md에 규칙 참조 추가
```

### 다른 프로젝트에서 ECC 활용하기

ECC 플러그인은 **글로벌 설치**이므로, 어떤 프로젝트 디렉토리에서 `claude`를 실행하든 자동으로 모든 스킬/커맨드/에이전트가 활성화됩니다.

```bash
cd ~/projects/bearwatch && claude    # ← ECC 자동 활성화
cd ~/projects/new-app && claude      # ← ECC 자동 활성화
```

## 요약

1. **스킬 명령어를 외울 필요 없습니다** — 자연어로 요청하면 ECC가 자동 활성화
2. **보안은 3중 자동 보호** — 코드 작성(에이전트) + 커밋(훅) + 리뷰(통합검토)
3. **코드리뷰 → 커밋 → PR** 순서로, 한 문장으로 요청 가능
4. **팀 병렬 작업** 시 각 에이전트가 독립 컨텍스트를 가져 메인 세션 보존
5. **`/save-session` → `/resume-session`** 으로 세션 간 맥락 유지
6. **`/handoff`** 로 작업 인수인계 문서 자동 생성
7. **플러그인이 이미 글로벌 설치**되어 있어, 모든 프로젝트에서 바로 사용 가능
